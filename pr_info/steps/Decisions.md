# Decisions Log — Issue #9 (CLI `init` and `verify`)

Round 1 plan-review decisions (supervisor + user). One bullet each, with rationale.

- **Q1 — `tomlkit` kept as mandatory runtime dep**: `init --pyproject` needs to round-trip `pyproject.toml` preserving formatting/comments; stdlib `tomllib` is read-only. Cheap (~100 KB), well-maintained. Already added to `pyproject.toml` runtime deps in the worktree.
- **Q3 — MSSQL EXPLAIN deferred until MSSQL backend lands (issues #5/#6)**: this PR's `verify` calls `backend.explain(sql)` only; MSSQL's current `NotImplementedError` surfaces as a clean `[ERR]` row with the exception message. No `SET SHOWPLAN_TEXT ON` plumbing now — that ships with the MSSQL backend itself.
- **Q5 — `tools_registered_count` = tools mounted for active backend**: not the total queries in `default_queries.toml`. `verify_builtin` calls the same mount-computation helper that `server.py` uses (extract one if needed), so the reported count matches what an MCP client would actually see. Adds a deliberate coupling between `verify` and server-mount logic, documented in `verify_builtin`.
- **I1 — Step 1 and step 2 kept separate**: rename-only (step 1) vs. field-shape change + sqlite fix (step 2). Each commits cleanly on its own; merging them would mix two reviewable concerns.
- **I2 — Step 3 split into 3a + 3b**: 3a lands the `cli/` package skeleton, dispatch refactor, tach + importlinter wiring; 3b copies `mcp_coder`'s `--help` infrastructure verbatim (`HelpHintArgumentParser`, `WideHelpFormatter`, `--version`, unified `help` subcommand) and finalizes all `help=` strings. Splitting keeps each commit focused and lets the help-polish reuse a stable scaffold.
- **C6 — `extract_sql_params` promoted to public in `schema_tools`**: previously private `_extract_sql_params`; verify is now the second consumer. Drop the underscore and update internal call sites. Formalizing the API is cheaper than reaching into a private name.
- **D1 — New CLI reference doc + existing-doc gaps fixed**: `docs/cli.md` covers `init` and `verify` (flags, example output, exit codes). Existing-doc fixes: `mcp-tools-sql.md` line ~859 (`connection_string=` → `path=`); `README.md` Quick Start (align with real `init`/`verify` UX; rename `--user-config` to `--database-config`); `docs/architecture/architecture.md` (note new `cli/` layer); `vulture_whitelist.py` (drop `_.connection_string` and `_.load_user_config`). Bundled into a new dedicated step 10.
- **D2 — Copy `mcp_coder`'s `--help` infrastructure verbatim, do not reimplement**: `HelpHintArgumentParser` and `WideHelpFormatter` come straight from `p_coder:src/mcp_coder/cli/parsers.py`; the `--help` / `--version` / `help`-subcommand wiring mirrors `p_coder:src/mcp_coder/cli/main.py`. No reinvention — keep both projects' UX consistent and avoid subtle behavioral drift.
- **I3 — Two separate `init` template builders**: `_build_project_template_standalone(backend)` and `_build_project_template_pyproject(backend)`. The two outputs have intentionally different content (full commented examples vs. minimal active-keys + comment pointer). A shared `_build_project_template(comments=...)` helper would obscure that intent.
- **I4 — Loader's own log warnings allowed to interleave with verify output**: no `logging.disable(...)` or logger-level adjustments inside `verify.run()`. Sensitive-key WARN from `load_query_config` may print alongside verify rows. Acceptable noise; suppressing it would risk hiding real diagnostic output.
- **I5 — Formatter includes `warn` symbol from step 5 onward**: `STATUS_SYMBOLS` contains `"warn" → "[WARN]"` from the step-5 scaffold. The actual `[WARN]` row is wired in step 7 (sensitive-key promotion), but the symbol entry sits in the map immediately so later steps don't have to touch the constant.
- **I6 — Keep "final compliance check" section in step 9**: the (i)–(xiv) coverage matrix stays as a visible reminder mapping issue tests to test files. Helps the reviewer confirm test completeness at a glance — small footprint, large signal.
- **`verify_connection` returns `(result_dict, open_backend_or_None)` from the start (step 7)**: M1 callers ignore the second element. M2 (step 8/9) reuses the open backend for EXPLAIN / INFORMATION_SCHEMA queries without reconnecting. Avoids restructuring the contract mid-stream.
- **SQLite EXPLAIN with parameters**: `_check_sql_explain` builds a dummy params dict from `params.keys()` mapped to type-appropriate placeholders (`""`, `0`, `0.0`, `datetime(2000,1,1)`) and passes them to `backend.explain(sql, dummy_params)`. SQLite's `EXPLAIN QUERY PLAN` requires bound values to compile parameterized SQL.
- **`tach.toml` declares all `cli.commands` import edges upfront in step 3a**: `backends`, `schema_tools`, `formatting`, `utils` all listed even though some only land in steps 5/6/7. Avoids touching `tach.toml` in every later step; keeps later commits surgical.

## Round 2 plan-review decisions (2026-05-01)

- **Q1 — `tools_registered_count` scope kept minimal (Option A)**: `tools_registered_count = len(load_default_queries())`. No helper extracted from `server.py`, no per-backend or disabled-query filtering. Real filtering is deferred to a follow-up PR — to be tackled when `server.py` actually introduces disabled-query / per-backend-override filtering. **Replaces** the Round 1 Q5 decision (which proposed extracting `compute_mounted_tools`); rationale: there is nothing to filter today, so introducing a shared helper would couple two layers prematurely. Aligns step 6 with the no-args `verify_builtin() -> dict[str, Any]` already in `summary.md`.
- **Step 8 `explain()` signature widening accepted**: both `DatabaseBackend.explain` (ABC, in `src/mcp_tools_sql/backends/base.py`) and `SQLiteBackend.explain` (in `src/mcp_tools_sql/backends/sqlite.py`) widen to `explain(self, sql: str, params: dict[str, Any] | None = None) -> str`. SQLite passes `params or {}` to the cursor (matches the existing Round 1 decision that EXPLAIN QUERY PLAN needs bound values). Backwards-compatible — default `None` means existing single-arg callers keep working. Without this, step 8's `_check_sql_explain` algorithm (which calls `backend.explain(sql, dummy_params)`) would not compile.
- **Step 7 try/finally lifecycle moved out of step 8 into step 7**: the open-backend `try/finally` that closes the backend after M2 sections finish belongs in step 7's orchestrator algorithm, since step 7 is where the open-backend path first appears (`verify_connection` returns `(result, open_backend_or_None)`). Step 8 just plugs the QUERIES section into the existing `try` block; step 9 plugs in UPDATES. Keeps lifecycle wiring in one place.
- **Step 6 `verify_builtin` signature stays no-args**: `verify_builtin() -> dict[str, Any]` (no `backend_name` argument) — aligns step 6 with `summary.md`. The orchestrator passes the backend name only to `verify_dependencies`. Drift between step 6 and the summary is fixed.
- **Step 4 `--output` ignored under `--pyproject` (no error)**: when `--pyproject` is set, the project block is written into `pyproject.toml`'s `[tool.mcp-tools-sql]` table and `--output` is ignored. If `--output` was non-default, log a `logger.debug(...)` line acknowledging it was ignored. No stdout warning, no error exit. Documented explicitly in step 4's `_run_pyproject` algorithm.
- **Step 3a `setup_logging` is the real one, not the placeholder**: `main.py` currently has a no-op local `_setup_logging`. Step 3a replaces it with `setup_logging` imported from `mcp_tools_sql.utils.log_utils`, called once in `main()` before subcommand dispatch with the parsed `--log-level` / `--log-file` / `--console-only` flags. Eliminates a silent dead-code path.
- **Step 9 schema-fallback documented as intended behavior**: `_list_table_columns` does **not** auto-default to `dbo` / `public`. Users must configure `schema = "..."` in their `[updates.<name>]` block when the table lives outside the connection's default schema. Empty `schema` → table lookup fails → `[ERR] Table not found` — verify reflects what was configured rather than silently substituting environment-specific defaults.
